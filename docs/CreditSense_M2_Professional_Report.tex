\section{Project Snapshot}

\subsection{What We Built}
CreditSense v2 is a conversational credit assessment system where a user enters borrower details, interacts with an AI assistant,
and receives a structured decision report supported by regulatory citations from an indexed RAG corpus.

\subsection{Delivered Milestone 2 Scope}
\begin{tabularx}{\textwidth}{>{\bfseries}p{0.28\textwidth}X}

Area & Implemented Capability \\
\midrule

Backend & FastAPI endpoints for health, state initialization, turn processing, seeded parameters, and ingestion \\
Orchestration & LangGraph node graph with deterministic routing \\
Retrieval & Chroma persistent store + ONNX MiniLM embeddings (via Chroma) \\
Decisioning & Deterministic policy thresholds + ML adapter score + risk flags \\
Reporting & English report generation + Hindi translation + PDF export \\
Deployment Assets & Procfile, render.yaml, runtime.txt (Python 3.11.0) \\
\bottomrule
\end{tabularx}

\section{Viva Voce Preparation}

\subsection{System Architecture (The How)}
The architecture is intentionally layered so each responsibility is clear and auditable:
\begin{enumerate}
  \item Presentation Layer: Streamlit UI captures borrower profile and user questions.
  \item API Layer: FastAPI validates payloads, normalizes inputs, and invokes graph turns.
  \item Orchestration Layer: LangGraph runs deterministic node transitions.
  \item Intelligence Layer: Retriever + policy + ML + LLM compose final analytical output.
  \item Output Layer: Bilingual report rendering and PDF export.
\end{enumerate}

\subsection{Code Snippet 1: Graph Routing Logic}
\begin{lstlisting}[style=pythonstyle,caption={Conditional routing in graph orchestration},label={lst:graph}]
def _after_conversation(state: AgentState) -> str:
    return (
        "rag_retriever"
        if state.get("profile_complete", False)
        and state.get("report_requested", False)
        else END
    )

graph.add_conditional_edges(
    "conversation",
    _after_conversation,
    {
        "rag_retriever": "rag_retriever",
        END: END,
    },
)
\end{lstlisting}

Why this matters:
\begin{itemize}
  \item Prevents report generation on incomplete borrower profile.
  \item Makes decision path deterministic and explainable to reviewers.
\end{itemize}

\subsection{Code Snippet 2: API Seed Parameter Normalization}
\begin{lstlisting}[style=pythonstyle,caption={Strong typing and normalization before scoring},label={lst:seed}]
def _normalize_seed_parameters(parameters: dict[str, Any]) -> dict[str, Any]:
    normalized: dict[str, Any] = {}
    for field in REQUIRED_FIELDS:
        if field not in parameters:
            continue
        value = parameters.get(field)
        if value is None:
            continue
        if isinstance(value, str):
            value = value.strip()
            if not value:
                continue
        if field in INT_FIELDS:
            normalized[field] = int(float(value))
            continue
        if field in FLOAT_FIELDS:
            normalized[field] = float(value)
            continue
        normalized[field] = str(value)
    return normalized
\end{lstlisting}

Why this matters:
\begin{itemize}
  \item Prevents malformed user input from corrupting downstream policy and ML logic.
  \item Ensures consistent numeric operations for DTI, EMI, and LTV calculations.
\end{itemize}

\subsection{Code Snippet 3: Deterministic Policy Scoring}
\begin{lstlisting}[style=pythonstyle,caption={Transparent score deductions based on check status},label={lst:policy}]
score = 100.0
for check in checks:
    status = check["status"]
    if status == "FAIL":
        score -= 18
        risk_flags.append(f"{check['metric']} failed")
    elif status == "WATCH":
        score -= 10
        risk_flags.append(f"{check['metric']} needs review")

if score >= 80:
    decision = "APPROVE"
elif score >= 60:
    decision = "CONDITIONAL"
else:
    decision = "ESCALATE"
\end{lstlisting}

Decision formula used in viva explanation:
\[
S = \max\left(0,\min\left(100,100-18F-10W\right)\right)
\]
where $F$ is number of \texttt{FAIL} checks and $W$ is number of \texttt{WATCH} checks.

\subsection{Code Snippet 4: Retrieval Query Composition}
\begin{lstlisting}[style=pythonstyle,caption={Profile-aware retrieval query for regulatory grounding},label={lst:rag}]
def build_query_from_profile(profile, ratios, conversation_context="") -> str:
    return " ".join([
        f"loan purpose {profile.get('loan_purpose', '')}",
        f"loan amount {profile.get('loan_amount_requested', 0)}",
        f"credit score {profile.get('credit_score', 0)}",
        f"post loan dti {ratios.get('post_loan_dti', 'na')}",
        f"collateral {profile.get('collateral_type', 'none')}",
        f"borrower conversation context {conversation_context[:400]}",
        "RBI NBFC underwriting fair practices",
    ])
\end{lstlisting}

Why this matters:
\begin{itemize}
  \item Retrieval is not generic; it is conditioned by borrower profile and financial ratios.
  \item The response can reference relevant regulatory context instead of unsupported text.
\end{itemize}

\section{Design Reasoning \& Justification}

\subsection{Why this specific architecture}
\begin{itemize}
  \item Regulated credit decisions require explainability; deterministic policy + graph routing was mandatory.
  \item User trust requires readable narrative; LLM is used for explanations, not sole decision authority.
  \item Milestone constraints favored rapid deployment and maintainability; Streamlit + FastAPI gave fastest reliable path.
\end{itemize}

\subsection{Why this model and retrieval stack}
\begin{itemize}
  \item ONNX MiniLM via Chroma chosen for practical retrieval quality under CPU-focused runtime constraints.
  \item Chroma chosen for local persistence and simple operational footprint.
  \item Groq-backed large model chosen for report language quality and Hindi translation quality.
\end{itemize}

\subsection{Why hybrid deterministic + ML + LLM}
A single-method system is brittle:
\begin{itemize}
  \item Pure deterministic: strong control, weak narrative intelligence.
  \item Pure LLM: strong language, weak auditability.
  \item Hybrid approach: preserves compliance explainability while keeping conversation/report quality high.
\end{itemize}

\section{Learnings / Key Takeaways}
\begin{enumerate}
  \item Architecture clarity matters more than model size in viva evaluations.
  \item Deterministic fallbacks are essential for production-like reliability.
  \item Good retrieval requires query engineering, not only vector DB setup.
  \item Clear state contracts simplify debugging in conversational AI systems.
  \item Visual documentation dramatically improves reviewer comprehension.
\end{enumerate}

\section{Rapid Viva Q\&A Drill}
\textbf{Q1: Explain data flow from input to output in 30 seconds.}\\
A: User enters profile in Streamlit, API normalizes and merges state, LangGraph runs guardrail and conversation. If profile is complete and report requested, retrieval fetches regulatory evidence, policy computes decision score, report and translation nodes generate bilingual outputs, and PDF export completes delivery.

\textbf{Q2: Why is your decisioning defensible?}\\
A: Decision score is deterministic with explicit deductions per failed/watch checks, and each decision band is transparent (APPROVE, CONDITIONAL, ESCALATE).

\textbf{Q3: What happens when LLM API fails?}\\
A: The system remains functional using deterministic fallback for chat, report generation, and translation.

\textbf{Q4: How do you prove regulatory grounding?}\\
A: Retriever pulls citation-ready chunks from indexed RBI/NBFC corpus conditioned by borrower profile and conversation context.

\textbf{Q5: If you had one week more, what would you improve first?}\\
A: Add authentication, observability traces, and external persistent vector storage for production hardening.
